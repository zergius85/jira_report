## Март 2026 — Dashboard 2.0

**Статус:** ✅ Реализовано (8 марта 2026, коммиты e9b3921, c3d9cb3, 459ed08, d494644, 9ea95a0, f282829)

### D2.0-1: Интерактивный UI

**Проблема:** Статичный интерфейс, неудобно для частого использования.

**Решение:**
- Быстрые пресеты: 4 кнопки (Прошлый месяц, Текущий месяц, Прошлая неделя, Эта неделя)
- Скрываемая шапка фильтров (экономия места)
- KPI карточки с трендами (сравнение с предыдущим периодом)
- Risk Zone виджет на дашборде
- Улучшенная сетка графиков

**Файлы:**
- `templates/index.html` — +682 строк (CSS, JS, HTML)

**Эффект:** Ускорение формирования отчётов в 3-4 раза

---

### D2.0-2: История отчётов

**Проблема:** Отчёты не сохраняются, невозможно сравнить периоды.

**Решение:**
- SQLAlchemy модели: ReportHistory, ReportComment, ScheduledReport, TelegramSubscription
- Сервис `core/report_service.py` для работы с историей
- API endpoints: `/api/reports/history`, `/api/reports/compare`, `/api/reports/<id>`
- Сравнение периодов с автоматическим расчётом трендов

**Файлы:**
- `core/models.py` — 4 модели БД
- `core/report_service.py` — сервис для работы с БД
- `scripts/init_db.py` — инициализация БД

**Эффект:** Возможность отслеживать динамику и сравнивать периоды

## Март 2026 — P2 задачи

**Статус:** ✅ Реализовано (8 марта 2026)

### P2-1: Экспорт в CSV

**Проблема:** Только Excel формат, не все любят xlsx.

**Решение:**
- Endpoint `/api/download/csv`
- ZIP-архив с CSV-файлами для каждого блока отчёта
- Формат CSV с разделителем `;` (удобно для Excel)
- Кодировка UTF-8

**Файлы:**
- `web/app.py` — endpoint `api_download_csv()`

**Эффект:** Дополнительный формат экспорта для пользователей

---

### P2-2: Расширенное кэширование

**Проблема:** Повторные запросы отчётов выполняются каждый раз заново.

**Решение:**
- Кэш для `/api/report` на 5 минут (только production)
- Использование существующего декоратора `@conditional_cache`
- Автоматическая инвалидация по таймауту

**Файлы:**
- `web/app.py` — добавлен `@conditional_cache(timeout=300)` к `api_report()`

**Эффект:** Ускорение повторных запросов в 5-10 раз

---

### P2-3: JQL Builder

**Проблема:** JQL собирается конкатенацией строк, нет валидации.

**Решение:**
- Класс `JQLBuilder` с методами для каждого условия
- Валидация всех параметров (проект, даты, статусы)
- Санитизация идентификаторов и строк
- Разрешённые поля для сортировки
- Флюент-интерфейс (цепочка вызовов)

**Файлы:**
- `core/jql_builder.py` — класс `JQLBuilder`
- `tests/test_report.py` — тесты `TestJQLBuilder` (14 тестов)

**Пример использования:**
```python
jql = (JQLBuilder()
    .project('WEB')
    .status_not_in(['Closed', 'Done'])
    .duedate_between('2024-01-01', '2024-12-31')
    .assignee_in(['ivanov'])
    .order_by('duedate', asc=True)
    .build())
```

**Эффект:** Безопасное и удобное построение JQL-запросов

## Март 2026 — P1 задачи

**Статус:** ✅ Реализовано (8 марта 2026)

### P1-1: Пагинация для больших отчётов

**Проблема:** Jira API ограничивает максимальное количество результатов до 5000 за запрос.

**Решение:**
- Добавлена функция `search_all_issues()` с пагинацией
- Разбиение запроса на батчи по 100 задач
- Логгирование процесса загрузки
- Обновлены вызовы `search_issues()` на использование новой функции

**Файлы:**
- `core/jira_report.py` — функция `search_all_issues()`
- `tests/test_report.py` — тесты `TestSearchAllIssues`

**Эффект:** Поддержка отчётов с любым количеством задач (>5000)

---

### P1-2: Унификация логирования

**Проблема:** В некоторых местах использовался `print()` и `traceback.print_exc()`.

**Решение:**
- Заменён `traceback.print_exc()` на `logger.error(exc_info=True)`
- Добавлены информативные сообщения об ошибках
- Единый стиль логирования во всех обработчиках ошибок

**Файлы:**
- `web/app.py` — замена в endpoints `/api/issue-types`, `/api/task-info`, `/api/report`

---

### P1-3: Бэкап конфигурации

**Проблема:** `.env` — единственное место хранения конфигурации.

**Решение:**
- Скрипт `scripts/backup-config.ps1` для PowerShell
- Автоматическое создание бэкапов с датой
- Хранение последних 10 бэкапов
- Автоматическое удаление старых бэкапов

**Файлы:**
- `scripts/backup-config.ps1`

---

### P1-4: Версионирование конфигурации

**Проблема:** Нет проверки совместимости `.env` при обновлении.

**Решение:**
- Добавлена переменная `CONFIG_VERSION=2.3` в `.env.example`
- Проверка версии в `core/config.py`
- Предупреждение в логе при несовпадении версий

**Файлы:**
- `.env.example` — `CONFIG_VERSION=2.3`
- `core/config.py` — проверка `REQUIRED_CONFIG_VERSION`

## Март 2026 — Безопасность (P0)

**Статус:** ✅ Реализовано (8 марта 2026)

### Санитизация JQL-запросов

**Проблема:** Пользовательский ввод напрямую подставлялся в JQL-запрос без проверки.

**Решение:**
- Добавлены функции `sanitize_jql_identifier()` и `sanitize_jql_string_literal()`
- Все входные данные (проекты, пользователи, типы задач, даты) проходят валидацию
- Недопустимые символы блокируются (защита от SQL injection-подобных атак)
- Добавлены тесты для функций санитизации

**Файлы:**
- `core/jira_report.py` — функции санитизации
- `tests/test_report.py` — тесты `TestSanitizeJqlIdentifier`, `TestSanitizeJqlStringLiteral`
- `.env.example` — обновлена документация SSL_VERIFY
