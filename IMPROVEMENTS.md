## Март 2026 — Исправление критических проблем (P0)

**Статус:** ✅ Реализовано (9 марта 2026)

### P0-1: Глобальное отключение SSL

**Проблема:** SSL отключался глобально через переменные окружения `REQUESTS_CA_BUNDLE` и `CURL_CA_BUNDLE`, что влияло на ВСЕ HTTP-запросы в процессе.

**Решение:**
- Удалено глобальное изменение переменных окружения
- SSL отключается только для подключения к Jira через `options={'verify': False}`
- Добавлен комментарий о безопасности

**Файлы:**
- `core/jira_report.py` — строки 53-57

**Эффект:** Безопасная работа с SSL, влияние только на Jira-подключение

---

### P0-2: Утечка памяти в rate limiter

**Проблема:** `RateLimiter` хранил все запросы в словаре без ограничения размера, что приводило к утечке памяти.

**Решение:**
- Добавлен параметр `max_clients=10000`
- Используется `OrderedDict` вместо обычного dict
- Добавлен метод `_evict_old_clients()` для LRU-вытеснения старых клиентов
- Добавлено удаление пустых клиентов после очистки старых запросов
- Метод `move_to_end()` для обновления позиции клиента в LRU

**Файлы:**
- `web/middleware.py` — класс `RateLimiter`

**Эффект:** Ограниченная память, стабильная работа при длительной эксплуатации

---

### P0-3: Отсутствие ограничения кэша проектов

**Проблема:** Кэш проектов `_project_cache` не имел ограничения по размеру, что могло привести к потреблению большого количества памяти.

**Решение:**
- Добавлено ограничение `_MAX_PROJECT_CACHE_SIZE = 1000`
- Используется `OrderedDict` с LRU-стратегией
- Добавлен метод `_evict_old_project_cache()` для вытеснения старых записей
- Метод `move_to_end()` для обновления позиции элемента в кэше

**Файлы:**
- `web/app.py` — кэш проектов (строки 131-180)

**Эффект:** Предсказуемое потребление памяти, стабильная работа

---

### P0-4: asyncio.run() в фоновом задании

**Проблема:** В `_execute_scheduled_report()` использовался `asyncio.run()` внутри фонового задания, что могло вызвать ошибку при наличии активного event loop.

**Решение:**
- Добавлена проверка на активный event loop через `asyncio.get_running_loop()`
- При наличии loop создаётся новый для блокирующего выполнения
- Обёрнуто в try-except для обработки ошибок отправки в Telegram
- Fallback на `asyncio.run()` если нет активного loop

**Файлы:**
- `core/scheduler.py` — метод `_execute_scheduled_report()` (строки 290-318)

**Эффект:** Стабильная работа планировщика без ошибок event loop

---

### P0-5: Некорректная регистрация middleware

**Проблема:** В `web/middleware.py` использовались декораторы `@app.before_request` и `@app.after_request`, но объект `app` не был импортирован, что вызывало ошибку NameError при импорте модуля.

**Решение:**
- Удалены декораторы `@app` из функций middleware
- Создана функция `init_middleware(app)` для регистрации обработчиков
- Функция принимает Flask-приложение и регистрирует все middleware
- Добавлен импорт `init_middleware` в `web/app.py`
- Вызов `init_middleware(app)` сразу после создания приложения

**Файлы:**
- `web/middleware.py` — функция `init_middleware()` (строки 285-312)
- `web/app.py` — вызов `init_middleware(app)` (строка 46)

**Эффект:** Корректная работа middleware, отсутствие ошибок импорта

### P2-1: Добавление type hints

**Проблема:** Отсутствие аннотаций типов в некоторых функциях усложняет поддержку и рефакторинг кода.

**Решение:**
- Добавлен импорт `typing` в `core/jira_report.py` (`Callable`)
- Добавлены type hints для `search_all_issues()` → `(jira: JIRA, jql: str, ...) -> List[Any]`
- Добавлен импорт `typing` в `web/app.py` (`Tuple, Any, Dict, List, Optional, Union, Callable`)
- Добавлены type hints для функций в `web/app.py`:
  - `_get_api_projects() -> Tuple[Any, int]`
  - `_get_api_assignees() -> Tuple[Any, int]`
  - `_add_user_to_assignees(user: Any, assignees_dict: Dict[str, str]) -> None`
  - `_get_assignees_from_issues(jira: Any) -> Tuple[Any, int]`
  - `conditional_cache(timeout: int = 300) -> Callable`
  - `validate_json_request(f: Callable) -> Callable`
- Добавлены type hints в `web/middleware.py`:
  - Импорты: `Response, Callable, Any`
  - `RateLimiter.limit() -> Callable`
  - `APIError.__init__(payload: Optional[Dict[str, Any]])`
  - `APIError.to_dict() -> Dict[str, Any]`
  - `ValidationError.__init__(field: Optional[str])`
  - `handle_api_errors(f: Callable) -> Callable`
  - `before_request_timing() -> None`
  - `after_request_timing(response: Response) -> Response`
  - `log_request_info() -> None`
  - `init_middleware(app: Any) -> None`

**Файлы:**
- `core/jira_report.py` — добавлены type hints
- `web/app.py` — добавлены type hints
- `web/middleware.py` — добавлены type hints

**Эффект:** 
- Улучшена читаемость кода
- Упрощён рефакторинг и отладка
- Лучшая поддержка IDE (автодополнение, проверка типов)
- Все 69 тестов проходят

### P2-3: JQL Builder

**Проблема:** JQL собирается конкатенацией строк, нет валидации.

**Решение:**
- Класс `JQLBuilder` с методами для каждого условия
- Валидация всех параметров (проект, даты, статусы)
- Санитизация идентификаторов и строк
- Разрешённые поля для сортировки
- Флюент-интерфейс (цепочка вызовов)

**Файлы:**
- `core/jql_builder.py` — класс `JQLBuilder`
- `tests/test_report.py` — тесты `TestJQLBuilder` (14 тестов)

**Пример использования:**
```python
jql = (JQLBuilder()
    .project('WEB')
    .status_not_in(['Closed', 'Done'])
    .duedate_between('2024-01-01', '2024-12-31')
    .assignee_in(['ivanov'])
    .order_by('duedate', asc=True)
    .build())
```

**Эффект:** Безопасное и удобное построение JQL-запросов
